      • assign_shifts.php

        This is the frame container for master_shifts.php and people.php. In other words, it provides the main interface for workshift assignment. Because of the frames and extensive javascript, it's prone to breakage, but often it's hard to debug and fix. To complicate matters, master_shifts.php has an auxilary javascript file, master_shifts.js. This file acts as a go-between for master_shifts and people, and does a lot of the work for people, as a matter of fact. people keeps a houselist on the side, which has the number of hours assigned so far, and is a sortable, clickable list.

        The idea here is that as things happen in master_shifts, functions here get called. The main one is offer_options, which will go through everyone in the people list and display them appropriately, depending on their compatibility with the shift. To that end there are a number of other functions, all called by offer_options.

        Things to watch out for: there are hacks for Central Office-assigned shifts (co) in here -- more in master_shifts. Also, the usual shift rating style (wanted/unwanted or numeric) issues are here.

      • backup_database.php

        Fundamental script for backup/restoration. Originally it was meant to be called by users. It's now include()d automatically by many other scripts, and as such, outputs less in some circumstances. Important scripts that it is include()d by include recover_backup_database.php. Scripts that use it for making routine backups include update_house.php and basic_consts.php. It should be used by workshift managers at the end of every semester, so that the house has a nicely named backup that it knows is the final tally for each semester. It's a good idea to email the workshift managers about this, since they often don't do it.

        It is used both for backup and restoration, since the process is largely the same. It does things very mechanically, just copying tables one by one. It does not use the MySQL feature create table like because that became available in version 4.1, although now version 4.1 is the one that is used.

        Warning! Backing up and restoring is unfortunately a fairly dangerous operation as regards privileges. The workshift manager could, for instance, restore a backup during an election, thus overwriting votes. recover_backup_database.php has information on measures taken to ameliorate this problem, but again, backups and restores always have a potential for danger.

      • basic_consts.php

        Crucial page for workshift managers to go to. Unfortunately, it looks pretty intimidating. A lot needs to be done to set up the semester, though, and I thought that it was best that it all be done on one page.

        The first part sets up absolutely essential information that several other pages need to get started. The second part is potentially dangerous, but necessary. Preference forms should be deleted in a new semester, as should assigned shifts. They're separated because workshift managers occasionally want to delete assigned shifts at the end of the semester, so that they can generate temp weeks. Finally, fines, hours, etc., should be deleted in the new semester. While preference forms and assigned shifts can be deleted in the last days of an old semester, fines, etc., cannot be deleted until the semester is finished and backed up.

        The third part is just about preference forms. There's some javascript trickery with wanted/unwanted versus the ratings style. Then there is the text that appears on the preference form. It comes in two places, at the very top (before the user has filled in any information), and right before the shifts. These texts have two complications -- they can be considered as html, and there are escape sequences that can go into the text. They are accessed and set using static_text functions.

        The submission page is not very pretty, but it's not supposed to be viewed that often. The main complication is the loop that does set_static to all the variables. This part doesn't need to be modified as things are changed in the intro page, and so it has to deal with lots of things, like booleans and static_text. Booleans are dealt with in a different loop that tests individually for them. There is a further complication, as always, with the rating system -- if we're in wanted/unwanted, we don't want to tell the user that we're just emulating wanted/unwanted with ratings between 0 and 2, so we need to check for that.

        Finally, the real modification of the database begins. First, we redo the weekly_totals_data table based on the number of weeks now in the semester and the default number of hours, using create_and_update_weekly_totals_data. Then, if we are going to delete things, we do it. First, the weekly sheets and fines. Deleting weeks is done using delete_weeks.php, and other stuff is done transactionally, to save something. The second two are both pretty simple -- empty out wanted_shifts and delete notes, submit_date, and zero out availabilities in house_info for deleting preferences, and deleting all names in master_shifts.
